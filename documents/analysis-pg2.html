Step 2 - find a first time-hypothesis

a dcf message can be divided into different bands: general info, minute, hour, day, weekday, month and year band. There are actually 60 different values/constellations for the minute band, 24 for the hour band, 31 for the day band and 12 for the month band. It seems most practicable to search each band as a whole, for all of  its possible values. While each minute band will be forwarded one step per minute to give a single one matching  band, the hours, day, month will be left without any forwarding as long as the time hypothesis is not yet clearly found. The user shall be given the advise better not to start the application at the end of an hour, since otherwise, the detection of a hypothesis might take some longer because the hour band will be set  forward undetected after minute 59..

Once the minute is found clearly, the hours can be forwarded one position as well after the 59th minute, as well as in the same manner  the days, months and years. Probable it will be best to use some kind of weighted average over several minutes to define the correlations ρ, so that any wrong value will fall out soon. Since the years can have rather a lot of values but very seldom a change during application run, their value shall be found on a one bit per correlation base and not as a band.

It has to be verified with PTB that they always use the same constellation of bits for any single value since there are more than one possibility for many values.

During a time hypothesis is searched, the application will also verify the 0-second position and δT. Therefor second 58,59, 0 and 1 will be recorded on a sec100 granularity with the 0-1 mask, the 0-second position will be tested within second 59 and 0. Any δT will be updated discretely and in case of a 0-second position error, step 0 and 1 will be repeated completely.

 

 

 

 

 

Step 3 - verify time-hypothesis

As soon as all the time bands are found clearly, statistics will be reset, and the hypothesis checked over a few minutes. Now all the timebands will be forwarded as needed. It is not yet clear, if the bands shall be checked for correlation to all its possible value or just towards the expected value. It is foreseen, to calculate the normal average values instead of the weighted values, with regualr statistics update, respectively setting average over n-values as average over much less values (eg. 1, 3 or 5) to give the current values more weight..... yet to be defined...

Again, the 0-second position and δT  will be checked/updated in the same way as in step 2.

 

Step 4 - time forwarding in real-time

After a short time of time-hypothesis verification, the time shall be forwarded in realtime to the user. This will be done in dcf coding at the same time, the code comes in since the application knows it in forward, but also it shall be transmitted as time string over RS232, USB.
